FASE 1: REGISTRAZIONE
---------------------
Frontend:
  Componente: pages/auth/register.component.ts
  Template:   pages/auth/register.html
  Style:      pages/auth/register.css

Backend:
  Controller: auth/controller/AuthController.java        (POST /api/v1/auth/register)
  Service:    auth/service/RegistrationService.java       (register())
  DTO:        auth/dto/RegisterRequestDto.java
  Model:      user/model/User.java

Flusso:
1. Il customer apre l'app e inserisce dati anagrafici:
   - Dati personali: nome, cognome, data nascita, genere, luogo/provincia nascita
   - Contatti: email, telefono, residenza
   - Documenti: codice fiscale, professione
   - Credenziali: email (usata come username), password
2. Frontend invia POST /api/v1/auth/register
3. Backend RegistrationService.register():
   a. Verifica unicità email e username (query su tabella users)
   b. Recupera ruolo "C" da tabella role_types
   c. Recupera stato "PENDING" da tabella user_statuses
   d. Crea oggetto User con tutti i dati
   e. Hash della password con SHA-256 (MessageDigest)
   f. Salva plain password temporaneamente (per provisioning Keycloak futuro)
   g. Salva utente in tabella users (stato PENDING)
4. NESSUN conto creato in questa fase
5. Frontend mostra messaggio di conferma, in attesa validazione dipendente

FASE 2: CONVALIDA REGISTRAZIONE (da parte del dipendente)
---------------------------------------------------------
Frontend:
  Componente: pages/employee/employee-registrations.component.ts
  Template:   pages/employee/employee-registrations.html

Backend:
  Controller: employee/controller/EmployeeUserController.java
  Service:    user/service/UserService.java               (validateRegistration(), rejectRegistration())
  Service:    auth/service/KeycloakAdminService.java       (createUser(), assignRealmRole())
  Service:    account/service/AccountService.java          (createInitialAccountForUser())
  Service:    card/service/CardService.java                (issueDebitCard(), activateCardsByAccountId())

Flusso:
1. Il dipendente accede al pannello admin
2. GET /api/v1/employee/users/registrations/pending mostra utenti PENDING
3. Backend UserService.getPendingRegistrations():
   - Query: SELECT * FROM users WHERE status_id = (id di PENDING)
   - Ritorna lista con: userId, firstName, lastName, email, registeredAt
4. Il dipendente sceglie di validare o rifiutare

SE VALIDATA (PUT /api/v1/employee/users/registrations/{userId}/validate):
  a. UserService.validateRegistration():
     - Verifica utente esiste e non è già ACTIVE
     - Chiama KeycloakAdminService.createUser():
       * Ottiene admin token da Keycloak (POST /realms/master/protocol/openid-connect/token)
       * POST /admin/realms/{realm}/users con username, password, email, nome, cognome
       * Estrae keycloakId dalla response Location header
       * Assegna realm role "C" all'utente Keycloak
     - Salva keycloakId nell'utente, cancella plain password
     - Imposta stato utente -> ACTIVE (tabella user_statuses)
     - Chiama AccountService.createInitialAccountForUser():
       * Genera IBAN univoco (IT + 15 chars UUID, max 5 tentativi)
       * Crea Account con balance=0, status=INACTIVE
       * Salva in tabella accounts
     - Imposta conto -> ACTIVE
     - Chiama CardService.issueDebitCard():
       * Recupera tipo DEBIT da card_types
       * Recupera stato ACTIVE da card_statuses
       * Genera numero carta casuale 16 cifre (verifica unicità)
       * Genera CVV casuale 3 cifre
       * Imposta scadenza = oggi + 5 anni
       * Salva in tabella cards
     - Se Keycloak fallisce: rollback, cancella utente Keycloak se creato
  b. Tabelle toccate: users, accounts, cards, account_statuses, card_types, card_statuses

SE RIFIUTATA (PUT /api/v1/employee/users/registrations/{userId}/reject):
  - UserService.rejectRegistration():
    * Verifica utente è in stato PENDING
    * Imposta stato -> ANNULLED
  - Tabelle: users

FASE 3: LOGIN
-------------
Frontend:
  Componente: pages/auth/login.component.ts
  Template:   pages/auth/login.html
  Service:    core/services/auth.service.ts               (login())
  Guard:      core/guards/auth.guard.ts                   (authGuard, roleGuard)
  Interceptor: core/interceptors/auth.interceptor.ts      (aggiunge JWT alle richieste)

Backend:
  Controller: auth/controller/AuthController.java        (POST /api/v1/auth/keycloak-login)
  Service:    auth/service/KeycloakAdminService.java

Flusso:
1. Il customer inserisce email e password
2. Frontend invia POST /api/v1/auth/keycloak-login
3. Backend:
   a. Verifica stato utente nel DB (deve essere ACTIVE)
   b. Se PENDING/ANNULLED/SUSPENDED -> errore 403 "Account not active"
   c. Autenticazione con Keycloak (verifica credenziali)
   d. Se credenziali errate -> errore 401
   e. Se valide: ritorna JWT access token
4. JWT contiene: ruolo (C/D), userId, limitsSetupComplete
5. Frontend salva token in localStorage
6. Ogni richiesta successiva include header: Authorization: Bearer {token}

FASE 4: PRIMO ACCESSO - CONFIGURAZIONE LIMITI
---------------------------------------------
Frontend:
  Componente: pages/customer/customer-limits-setup.component.ts
  Template:   pages/customer/customer-limits-setup.html
  Service:    core/services/customer.service.ts           (getLimits(), setLimit())

Backend:
  Controller: account/controller/AccountController.java   (GET/PUT limits)
  Service:    account/service/AccountLimitService.java    (getLimits(), setLimitAsCustomer())

Flusso:
1. Frontend controlla limitsSetupComplete dal JWT
2. Se false: reindirizza a /customer/limits-setup
3. GET /api/v1/customer/accounts/{accountNumber}/limits mostra limiti attuali
4. Backend AccountLimitService.getLimits():
   - Query: SELECT * FROM account_limits WHERE account_id = ?
   - Per ogni limite: limitType, maxAmount, changePolicy
5. Il customer può modificare i limiti senza restrizioni (prima volta)
6. PUT /api/v1/customer/accounts/{accountNumber}/limits/{limitType}
7. Backend AccountLimitService.setLimitAsCustomer():
   - Verifica proprietà conto
   - Verifica changePolicy (prima configurazione: ignora restrizioni)
   - Validazione range:
     * ATM_WITHDRAWAL: min 10, max 300
     * POS_SPENDING: min 0.10, max 2500
     * SINGLE_TRANSFER: min 1, max 10000
     * INSTANT_TRANSFER_SINGLE: min 1, max 5000
     * DAILY_TRANSFER: min 1, max 15000
     * MONTHLY_TRANSFER: min 1, max 50000
   - Salva o aggiorna record in account_limits
8. Dopo conferma, limitsSetupComplete -> true
9. Ora può accedere alla dashboard

FASE 6: OPERAZIONI FINANZIARIE
-------------------------------
Frontend:
  Transazioni: pages/customer/customer-transactions.component.ts
  Service:     core/services/customer.service.ts

Backend:
  Controller: transaction/controller/TransactionController.java
  Service:    transaction/service/TransactionService.java

6.1 Deposito (POST /api/v1/customer/transactions/deposit):
  - TransactionService.deposit():
    * Verifica proprietà conto
    * Verifica conto è ACTIVE
    * transferFunds(null, account, amount, "DEPOSIT", "COMPLETED")
      - account.balance += amount
      - Salva in accounts
      - Crea record in transactions con sourceBalanceAfter=null, destBalanceAfter=nuovo saldo
  - Tabelle: accounts, transactions

6.2 Prelievo (POST /api/v1/customer/transactions/withdraw):
  - TransactionService.withdraw():
    * Verifica importo >= 10€ (MIN_WITHDRAWAL)
    * Verifica proprietà conto
    * accountLimitService.validateWithdrawal():
      - Cerca limite ATM_WITHDRAWAL in account_limits
      - Se non trovato, usa default MAX_VALUES["ATM_WITHDRAWAL"] = 300
      - Se importo > limite -> errore LIMIT_EXCEEDED
    * transferFunds(account, null, amount, "WITHDRAWAL", "COMPLETED")
      - account.balance -= amount
  - Tabelle: accounts, transactions, account_limits

6.3 Bonifico (POST /api/v1/customer/transactions/transfer):
  - TransactionService.transfer():
    * Verifica importo >= 1€
    * Se beneficiaryId fornito:
      - BeneficiaryService.resolveAccountNumber() risolve IBAN dal beneficiario
    * Verifica source != destination
    * Verifica proprietà source account
    * Verifica destination account esiste
    * accountLimitService.validateTransfer():
      - SINGLE_TRANSFER: importo singolo max 10000
      - DAILY_TRANSFER: somma bonifici oggi + importo <= 15000
      - MONTHLY_TRANSFER: somma bonifici mese corrente + importo <= 50000
      - INSTANT_TRANSFER_SINGLE: se istantaneo, max 5000
    * Se istantaneo (isInstant=true):
      - transferFunds(source, dest, amount, "INSTANT_TRANSFER", "COMPLETED")
      - Esecuzione immediata
    * Se programmato (isInstant=false):
      - Verifica scheduledDate > oggi e <= oggi+30 giorni
      - Crea transazione PENDING con scheduledDate
      - Sarà eseguita da executePendingTransfers()
  - Tabelle: accounts, transactions, beneficiaries, account_limits

6.4 Esecuzione Bonifici Programmati (background):
  - TransactionService.executePendingTransfers():
    * Query: SELECT * FROM transactions WHERE status_id = PENDING
      AND scheduled_date <= NOW()
    * Per ogni transazione:
      - Verifica conto sorgente è ACTIVE
      - Verifica conto destinazione è ACTIVE
      - Verifica fondi sufficienti su sorgente
      - Se tutto OK: status -> COMPLETED
      - Se KO: status -> FAILED con motivo
  - Tabelle: transactions, accounts

Photon API (https://photon.komoot.io)
  Descrizione: Servizio di geocoding open-source per autocomplete indirizzi
  File:
    Frontend: pages/auth/register.component.ts (onResidenceSearchInput(), selectAddress())
    Template: pages/auth/register.html (campo residenza con autocomplete)
  Installazione: Nessuna (API pubblica, no key richiesta)
  Funzionamento:
    - API REST basata su OpenStreetMap/Nominatim
    - Cerca indirizzi reali in Italia e nel mondo
    - Ritorna coordinate GPS, via, numero civico, CAP, città
    - Gratuita, no limiti di rate (ma raccomandato debounce)
  Utilizzo nel progetto:
    1. L'utente digita almeno 2 caratteri nel campo "Residenza"
    2. Frontend invia GET a:
       https://photon.komoot.io/api?q={query}&limit=5&countrycode=IT
    3. Risposta JSON con array di "features":
       {
         "features": [{
           "properties": {
             "street": "Via Roma",
             "housenumber": "15",
             "postcode": "00185",
             "city": "Roma",
             "country": "Italia"
           }
         }]
       }
    4. Frontend estrae: street, housenumber, postcode, city
    5. L'utente seleziona un suggerimento
    6. I campi vengono compilati automaticamente
    7. Alla registrazione, i dati vengono concatenati:
       residence = "Via Roma, 15, 00185"
  Codice frontend:
    const url = `https://photon.komoot.io/api/?q=${query}&limit=5&countrycode=IT`;
    fetch(url).then(r => r.json()).then(data => { ... })
  Note:
    - debounce di 100ms per evitare troppe richieste
    - countrycode=IT filtra solo risultati italiani
    - limit=5 massimo 5 suggerimenti
    - API pubblica, no autenticazione necessaria

Backend Comuni API (/api/v1/comuni)
  Descrizione: Endpoint per ricerca comuni italiani (luogo di nascita)
  File:
    Frontend:   pages/auth/register.component.ts (onCitySearchInput(), selectCity())
    Backend:    comuni/controller/ComuniController.java
    Service:    comuni/service/ComuniService.java
    DTO:        comuni/dto/ComuneDto.java
    Dati:       resources/comuni.json (~7900 comuni)
  Installazione: Nessuna (dati statici nel backend)
  Funzionamento:
    1. Al startup, ComuniService carica comuni.json dal classpath
    2. comuni.json contiene TUTTI i comuni italiani (~7900)
    3. Ogni comune ha: nome, sigla provincia, codice catastale
    4. Ricerca in memoria con due fasi:
       a. Prima cerca comuni che INIZIANO con la query (startsWith)
       b. Poi cerca comuni che CONTENGONO la query (contains)
    5. Massimo 10 risultati
  Utilizzo nel progetto:
    1. L'utente digita almeno 1 carattere nel campo "Luogo di nascita"
    2. Frontend invia GET a:
       http://localhost:8081/api/v1/comuni?search={query}
    3. Risposta JSON:
       [{"nome":"Roma","provincia":"RM","codiceCatastale":"H501"}, ...]
    4. L'utente seleziona un comune
    5. Si compilano: birthPlace, birthProvince, birthCatastalCode
    6. Il codice catastale serve per calcolare il Codice Fiscale
  Struttura comuni.json:
    [
      {"nome": "Roma", "sigla": "RM", "codiceCatastale": "H501"},
      {"nome": "Milano", "sigla": "MI", "codiceCatastale": "F205"},
      ...
    ]
  Note:
    - Dati caricati una volta sola all'avvio (in-memory)
    - Ricerca O(n) ma sufficiente per ~7900 comuni
    - Codice catastale necessario per calcolo Codice Fiscale
    - Il frontend usa i risultati anche per il calcolo automatico CF

Calcolo Codice Fiscale (frontend):
  Descrizione: Generazione automatica del Codice Fiscale Italiano
  File:
    Frontend: pages/auth/register.component.ts (computeFiscalCode(), computeCIN())
  Funzionamento:
    1. Raccoglie: cognome, nome, data nascita, sesso, comune (codice catastale)
    2. Calcola 15 caratteri parziali:
       - Cognome: prime 3 consonanti (poi vocali, poi X)
       - Nome: consonanti 1,3,4 (se >3 consonanti) altrimenti come cognome
       - Anno: ultime 2 cifre dell'anno
       - Mese: lettera A-E-H-L-M-P-R-S-T (1-12)
       - Giorno: gg se maschio, gg+40 se femmina
       - Comune: codice catastale (6 caratteri)
    3. Calcola CIN (16° carattere) con algoritmo specifico:
       - Somma valori per posizioni pari/dispari
       - Resto 26 -> lettera A-Z
  Note:
    - Calcolo 100% lato frontend
    - Usa codice catastale dal backend comuni
    - Validazione: 16 caratteri esatti

TABELLA: role_types

  id: INTEGER PK, user_status: VARCHAR(30) UNIQUE NOT NULL

  id: BIGINT PK, keycloak_id, username, first_name, last_name,

  limits_setup_complete BOOLEAN DEFAULT FALSE

TABELLA: account_statuses
  id: INTEGER PK, status_name: VARCHAR(30) UNIQUE NOT NULL
  Valori: INACTIVE, ACTIVE, FROZEN, CLOSED

TABELLA: accounts
  id: BIGINT PK, account_number (IT+15chars), balance, status_id FK,
  user_id FK (CASCADE), created_at, closed_at, source_account_number,
  initial_amount, version (optimistic locking)

TABELLA: limit_types
  id: INTEGER PK, limit_name, notes, change_policy (BANK_ONLY/USER_LOWER_ONLY/USER_FULL)

TABELLA: account_limits
  id: BIGINT PK, account_id FK (CASCADE), limit_type_id FK,
  max_amount, updated_at, UNIQUE(account_id, limit_type_id)

TABELLA: card_types
  id: INTEGER PK, type_name: DEBIT, CREDIT

TABELLA: card_statuses
  id: INTEGER PK, status_name: INACTIVE, ACTIVE, BLOCKED

TABELLA: cards
  id: BIGINT PK, card_number (16 cifre), card_type_id FK, holder_name,
  expiration_date (oggi+5anni), cvv (3 cifre), status_id FK, account_id FK (CASCADE)

TABELLA: transaction_types
  id: INTEGER PK, type_name: DEPOSIT, WITHDRAWAL, TRANSFER, SCHEDULED_TRANSFER, INSTANT_TRANSFER, INITIAL_TRANSFER

TABELLA: transaction_statuses
  id: INTEGER PK, status_name: PENDING, COMPLETED, FAILED, REJECTED, CANCELLED

TABELLA: transactions
  id: BIGINT PK, amount, type_id FK, status_id FK, description,
  source_balance_after, dest_balance_after, created_at,
  source_account_id FK (SET NULL), destination_account_id FK (SET NULL),
  scheduled_date

TABELLA: beneficiaries
  id: BIGINT PK, user_id FK, destination_account_number,
  destination_holder_name, nickname, created_at,
  UNIQUE(user_id, destination_account_number)
